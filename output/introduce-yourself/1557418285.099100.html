# Pete Vilter 2019-05-09 09:11
Hi folks! I'm Pete. My background is as a fullstack developer, but I've become very interested in databases, since a lot of fullstack app development comes down to loading and saving data from the DB. I've been working on https://github.com/vilterp/treesql, a database which has GraphQL-inspired tree-structured queries, and pushes out updates to those queries when the underlying data changes.

I work at https://www.cockroachlabs.com/, where I'm learning the craft of database architecture from some of the greats. A few years ago, I also made some contributions to the Elm time-travelling debugger (https://github.com/elm-lang/elm-reactor/pull/115), so I have a soft spot for that project :slightly_smiling_face:

# Pete Vilter 2019-05-14 21:48
Hi Drew,
re: what type of database, from the readme:
> relational (has tables with typed columns and foreign keys)

re: tree-structured queries; they are GraphQL-style, but with a more SQL-like syntax. Here's an example from the readme, slightly reindented:
MANY blog_posts {
  id, title, body,
  comments: MANY comments {
    id, body,
    author: ONE users {
      name
    },
  }
}

…and yes, as shown in the readme, the response is a whole tree as JSON.

re: how to do live queries: this code path: :stuck_out_tongue: https://github.com/vilterp/treesql/blob/master/pkg/live_queries.go#L94-L101
Basically, when a live query is first run, it saves "listeners" in a special data structure in memory. Later, when writes come in, we look up those listeners, find live queries interested in that write, and push it out to them.

There's nothing that makes it fundamentally easier to do that in a document store than in a relational database. CockroachDB (where I work) is built on top of a key-value store, as are document DBs, and we have one of the building blocks of live queries: changefeeds (https://www.cockroachlabs.com/docs/stable/create-changefeed.html), a way of getting a feed of writes going into the DB. The tricky part is mapping these writes to the queries that would be interested in them — i.e. "a comment with blog post id 5 was inserted => these 11 queries covered blog post 5, so they'd want to know about this"
