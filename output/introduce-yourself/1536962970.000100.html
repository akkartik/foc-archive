# Kartik Agaram 2018-09-14 15:09
Hi everyone! Lately I tend to introduce myself as a barbarian programmer. I mean that in a few different ways:

• I tend to be a curmudgeon, happier by myself than "organizing" into groups to achieve some shared goal. This doesn't mean I dislike collaboration. I just think we tend to conflate collaboration with synchronous collaboration, and asynchronous collaboration is a better fit for many tasks. Asynchronous collaboration can span great distances in space and time. You reading Aristotle could be asynchronous collaboration -- if you are then able to build on what you read in some way and put out some artifact that someone else can collaborate with. UX people like to say "don't make me think". I like to say "don't make me talk to you". Show me what you built in a form that I can read, listen to, play interactively with and gradually understand/appreciate. I may still come talk to you if the situation merits it (I actually love talking to people, believe it or not), but give me the choice.
• I think we've been doing city-living cheek-by-jowl for so long that we've forgotten what benefits it has, and continue cargo-culting it long after it's become superfluous. I don't mean that literally; I love living in cities. But our software stacks have many of the same patterns as cities, and they aren't really necessary: a) promiscuously adding dependencies, b) relying implicitly on some sort of division of labor, so that if we find a problem in a dependency we don't feel empowered to fix it for ourselves, instead turning to ways to try to get others to do it for us like paying, bullying, cajoling, passing laws, etc. It's worth swinging that pendulum back a bit, to imagine we're out on the steppe with just what we have on our backs and nobody else around for miles, and try to achieve our ends with 10x fewer libraries and frameworks. (More details: https://news.ycombinator.com/item?id=16882140#16882555 .)
• It emphasizes sustainability. When you live off the land (I gather; I wouldn't actually know) you are more attuned to how you affect your environment. Your externalities. We city dwellers just expect someone else to cart our trash out for us.

Unfortunately this is all aspirational, because the way we write software doesn't really work for a barbarian. I've been working very slowly on extricating a kernel of software and way of writing software that can be more self-sufficient and sustainable. I started out building a new programming language based on Lisp (http://akkartik.name/post/wart, which itself grew out of a startup; those who can't startup shave yaks), then grew frustrated with the stack below me and simultaneously realized languages don't matter, and build a toy VM and OS (https://github.com/akkartik/mu#readme ; http://akkartik.name/about), used it to teach programming for a couple of years, then grew frustrated with how slow it was and gradually faced up to the need to understand even more low-level details. Lately I've been building simple tools for programming in machine code (https://github.com/akkartik/mu/tree/master/subx#readme), but in the past week I've set this aside to design a type-safe language that I can bootstrap out of machine code (no link yet).

There's a reason I didn't say "break", and added the caveat about what "not lasting" means.

As a concrete example, simple tasks like copying strings, syncing to the file system and waiting on multiple file descriptors are fundamentally broken/insecure/errorprone on C. This stuff is an ongoing cost we all pay to this day, regardless of whether we program in C.

Exhibit B: the accretion of ruby/gem/rvm/bundler and python/distutils/pip/virtualenv. It happens because we feel disempowered to change the foundations. And yet each level up provides only temporary relief, and creates its own issues that need escaping. Escape lies downward.
