# Vladimir Gordeev 2020-03-31 14:49
@Daniel Garcia Maybe slack comment is not the best place to put it. I'll share what I can get from the top of my head.

Some of what I am writing here I have shared in my presentation: https://www.youtube.com/watch?v=t8UH3DjvxDE&t=37s

Some general principles:

1. Too much expressiveness is bad -- turns programming into Chaos (like Lisp macro hell). Too much restriction is also bad -- turns programming into a prison of Order (try to do complicated IO in Haskell). To get comfortable PL we need to walk a fine line between these Order and Chaos, to balance PL design. Many design decisions that I make just shift balance of Order and Chaos to make users less overwhelmed.
2. Programmers develop software through observe-modify-repeat loop in their heads. To make programming easy and efficient we need to optimize this loop. Make observation of the change fast, make modification safe and painless.
Some technical ideas:

1. Actors with isolated memory and communication by copying messages (from Erlang) -- brilliant idea that makes programming a lot easier. When it is guaranteed that this piece of code is influenced by these several inputs it is a lot easier to debug and reproduce it. This is absolutely necessary and should be as widespread as use of pure functions.
2. Application is basically a composition of many actors that message each other. During execution some actors might be spawned and killed as needed. This is very powerful and possible compositions might be quite overwhelming. To shift this situation towards Order I want to restrict possible compositions: it should be a regular tree.
3. One powerful data structure for almost everything: DAG. Pattern matching helps a lot with all kinds of manipulation. If an actor provides an interface of a DAG, it should be possible to pattern match against it. This way you can walk database, filesystem or web service in similar manner you would walk local DAG.
4. Test cases of input and expected output are attached directly to the function, it is part of the annotation. This makes it very easy to modify it -- tests for this particular function would run over and over. Precondition and postcondition code is part of the function. They help to distinguish bad input and error in the code, which helps to make IDE convenient. This feature is a lot easier to implement for structure-based source code.

# Steve Dekorte 2020-04-01 10:08
A coroutine is a lightweight cooperative thread. An example use might be waiting on asynchronous operations (like i/o) without loosing the state you’ve built up in your call stack - as you’d end up having to construct an even more complex (and likely inscrutable) state machine otherwise. So when you make your call to wait on an i/o operation, behind the scenes it sets aside your current stack and continues some other coroutine (or returns to the event loop). When the i/o operation event occurs, that callback resumes (switches to) the previously paused coroutine’s stack. In this way, you can writing asynchronous code as if it were synchronous, while still being aware of where those points are via things like naming conventions on methods that may pause the coroutine. You can also yield coroutines at safe points so they can do long running computations with stacks.

# Vladimir Gordeev 2020-04-07 00:55
Hey @Kartik Agaram!

The trees that are shown in the video were stored as a DAG with hashes as keys (just like in Git), if same tree node is used in several places, only one instance is stored. However, for now this reuse is not displayed in UI in any way.

Actually, I am not sure I understand about which benefits exactly you are talking. I have following thoughts in mind:

---

It is good to show that some subtree is used somewhere else. This way you can explore how this or that function is used in other places.

For example we have following DAG:

place1:
  subtree1: {foobar: baz}
place2:
  subtree2: {foobar: baz}

We can somehow display connection between subtree1 and subtree2 and navigate from one to the other.

However, if user decides to modify place1/subtree1, then editor should allow it for the sake of good user experience. It should feel like text and should not break things in other parts of the system by default. Thus, this connection between subtrees will be broken, because they are not equal anymore.

To have subtrees that are actually pointing to one thing and change simultaneously I plan to use explicit references to a subtree, for example:

references-list-keyword:
  ref1: {foobar: baz}
place1:
  subtree1: {reference-keyword: ref1}
place2:
  subtree2: {reference-keyword: ref1}

Tree editor should recognize this referencing and display references to the trees as actual trees (with some highlighting that would indicate referencing).

This referencing can be made nested, which would start to look like lexical scoping in many PLs.

For now DAG structure is not used for anything except storage saving, yes. Differences from simple trees may show up on the level above, in PL.

For example you have two different functions that expect trees with two different interfaces:
func1(Value):
  {title: S} = Value
  do_something(S)

func2(Value):
  {asString: S} = Value
  do_something(S)

Because we work with dags we can create a value that satisfies both interfaces with single value:
asString: "a string value"
title: "a string value"

Or, because it is a DAG, we can display it like that:
asString:
   title: "a string value"

Two keys, but one value.

That kind of thing will be really useful only on PL level, which I haven't reached yet.
