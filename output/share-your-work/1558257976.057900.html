# Adrian Sieber 2019-05-19 02:26
Hi folks!

I developed following webapp a few years ago and although I was (and still am) quite excited about the possibilities, I never released it. I couldn't imagine the programming community at large get excited about it. Also I was sick of preaching things nobody else seems to care about.

But now that I discovered this community, maybe a few of you might like it:

https://syvis.surge.sh/ErZoPhsMTcDvGT9pZswo/
Feel free to try it out with other JavaScript code. It completely runs in your Browser!

My thought process was that -- despite their flaws -- text editors are a quite powerful input method once you mastered the keyboard.
Consuming / investigating code in a text editor, however, not so. Our syntax highlighting, code style guides, indentation based code, etc. are just a desperate attempt to mitigate this.

What if coding was more like writing a LaTeX paper? The LaTeX code is good to build the document, and for reviewing and distributing it, the rendered PDF is much better suited.

Next steps could be:

- Custom themes, so that you can render the code to you liking
- Enable interactive editing for minor things like changing a number or a string, reordering parts, …

Here is another abstract from a slightly different angle:

In most programming languages
the semantic meaning of a program is disconnected from
the formatting of the code and the naming of variables and keywords.
This means semantically equivalent code can be written in a vastly
different textual representations.
This poses problems in terms of readability and understandability
of code.
Even minor modifications of punctuation or formatting in comparison to
a developers familiar style can drastically decrease the performance.
In order to mitigate these problems I propose to visualize code
in a graphical and deterministic way.
Instead of relying on the author of the code to format it in a acceptable
manner, each developer can choose a visualization of their liking.
This has the potential to decrease the error rate and
increase the development speed of programmers.

Looking foward to you feedback! :grin:

# Jules Jacobs 2019-05-30 12:20
One could argue that editing LaTeX code but looking at rendered pdfs is a flaw not a feature. LaTeX editors try to mitigate this by live rendering the document on the side, but this is still not ideal, because your cursor is on the left in the code and you're looking at the rendered pdf on the right. Some editors even show the cursor position in the rendered pdf. In the limit this seems to converge to showing all relevant information as an overlay in the rendered pdf, including the document tree, making the code view on the left side redundant.

If you have a code viewer that displays code in a superior way, then it seems to me that you'd want to use that view while editing as well.
