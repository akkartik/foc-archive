# JP Posma 2021-03-31 16:22
Yeah, agreed on all of that, and thanks @Jack Rusher! The blanket statements also concern me. I’m personally just interested in applications that really push the boundaries of performance, e.g. the last project I worked on was webviz.io.

There we ran into the limitations of a language like JS — we ended up doing a lot of manual memory management to avoid the overhead of a garbage collector, of non-zero-cost object instantiation and function calls, of DOM elements being fairly expensive when you get into the 10,000-100,000 range, and so on. It was very clear that what we were trying to do did not fit the language well — we ended up with tons of ArrayBuffers and manual operations on them, which was a lot faster, but pretty inconvenient.

So I’m interested in GUI systems in languages where you don’t have all these constraints, and where things like GC/refcount/dynamic dispatch/etc are opt-in, but where the default is much more low-level. This kind of programming is certainly not for every use case, but it seems strange to me that there really isn’t much out there to cater to these more extreme use cases, because surely it can’t be that rare…?
# JP Posma 2021-03-31 16:28
Also, in an application like that, it’s not like you can isolate just one part of your application that is the most performance critical — I wish that were the case! The core data that we were operating on easily ran into the many gigabytes, and since pretty much everything builds on top of that and needs to iterate over many objects, there are no easy abstractions. This is probably a pretty extreme use case though. :slightly_smiling_face:
