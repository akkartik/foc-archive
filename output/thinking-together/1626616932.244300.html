# Denny Vrandečić 2021-07-18 17:52
I don't think it is possible. In chemistry, you know what your elements are. In coding, you don't, there could be many different ones. You could just name all your functions based on a lambda calculus like SKI and then name them according to their implementation, but almost every function of interest can be implemented in multiple ways. And there is no (general) way to figure out if two implementations are equivalent.

Put differently: maybe it is possible, but if it is, we haven't figured out the foundational science needed to do something like IUPAC.

What one could do is to have names that are not meaningful and build a catalog of IDs to functions, but that's very different from IUPAC.

    I thought about Hungarian notation as well, but I don't think it is similar. The prefix tells you the logical or physical type of a variable, but it doesn't tell you the semantics, it doesn't fully define it.
    # Doug Moen 2021-07-26 14:59
    When I work on a large project, I accumulate a list of common words that describe recurring concepts, and join these together like morphemes to form identifiers. Lots of software projects do this, but the "morphemes" tend to be project specific, and don't tend to be particularly standardized. Although you can reuse morphemes from the standard library of the programming language you are using as your base.
    # Doug Moen 2021-07-26 15:04
    IUPAC names use fully standardized "morphemes" to fully describe chemical compounds. The closest I can think of to that situation is languages in the APL family (APL, J, K) where clusters of verbs and adverbs (single characters in APL and K) are used as idioms to denote various useful functions. These character clusters are the same length as library function names in conventional programming languages, but they fully describe the semantics of the function.
