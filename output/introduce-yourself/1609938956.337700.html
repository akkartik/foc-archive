# Ash Nehru 2021-01-06 05:15
Hello all ! @Tomas Petricek directed me here, thank you Tomas !
My current job, which I've been doing for just under 20 years, involves using a mixture of C++, python, and various shader languages (GLSL, HLSL, DirectX compute et al), that runs on clusters of CPU+GPU machines. Over the years I've become increasingly frustrated with the (continually worsening) complexity and poor productivity of the tools we use to write such applications, so I decided to do something about it.
For the last year, I've been working on a side-project called 'zero' - a programming language (and tool-chain) to let me write efficient, compiled, type-checked code that runs on the CPU and GPU, distributes across clusters naturally, but also gives me a proper live-coding workflow. Not a big ask, obviously :sweat_smile:
I've got to the stage where I finally have a reference interpreter that lets me experiment with language ideas and workflows, and I'm about to start work on the full-on compiler. I'm in the fortunate position of being able shortly to transition to working on the language full-time, with a target of starting a new company in early 2022.
Since I have an engineering rather than an academic background, I have a lot of learning to do - I definitely need some mentorship, guidance, critique, encouragement and moral support. Hopefully I can also be of some use to others as well !

# taowen 2021-01-06 20:39
One thing I find it is absurd is julia allocate memory in GPU, tracked by a variable in CPU. When garbage collector running in the CPU find the variable out of scope, the corresponding GPU memory will be deallocated. When a variable represent both a chunk of memory in CPU and in GPU, it complicates the programming reasoning. It would be great if you can address the issue. Instead of assuming every line of code runs in the CPU, we need a programming model to fully drive computation from GPU, instead of being driven.
