# Kartik Agaram 2019-10-13 13:25
> because if your users only understand things from the x86 level on up, they don't really understand things, imo

Hmm, not sure why you think this. You have to accept some substrate.

I'm not a fan of x86, but it is easily accessible for anyone who wants to try Mu out on a whim.

> Would it be correct to say Mu is a language that compiles to x86 Assembly, with the goal of being easier to use than direct x86, but not to be far removed from x86 so that the programmer is at all times aware of what the x86 machine is doing?

Yep! The Mu programmer can always bump down to machine code at a moment's notice, no matter how many levels of abstraction it gains over time. And all the tooling takes this capability for granted, so it can be less hermetically sealed and need less code at all levels of the stack.

If conventional stacks try to be like the Starship Enterprise, all gleaming chrome, Mu is designed like Serenity, with the ductwork hanging out.

# Edward de Jong 2019-10-13 14:21
Nothing wrong with picking intel. They make good chips and it is a simple enough chip to work with. But nowadays every shipping chip includes the AMD x64 extensions, which AMD licensed back to Intel. It adds enough registers so that you aren't constantly shuffling data in and out of the measly 2 registers EAX and EBX. Remember that multiply uses the ECX EDX registers, so intel x86 really is a 2 general purpose register machine, and a huge percentage of instructions are loading and storing data from those registers. Another thing to consider is that most of the work Intel has been doing in the last 5 years with their instruction set is adding crazy instructions that have zero actual practical value, but are solely designed to confound and raise costs for the cloners in China, who have caught up with the basic instruction set, and sell cut-rate intel clone chips. The only useful instruction intel has added in the last 10 years is a true random number generator, which uses electron noise conditioned to produce the first actual random numbers for computers. Part of the AES extensions. But i would ignore the batshit crazy instructions like the GF2P8AFFINEINVQB — Galois Field Affine Transformation Inverse, which not one person in a million will ever use, much less understand.
# Kartik Agaram 2019-10-13 14:24
IMUL actually uses EDX:EAX (https://c9x.me/x86/html/file_module_x86_id_138.html). If we all stopped using EAX (the accumulator register) because of that, then x86 would not have taken over the world.

In the 40kLoC of x86 machine code I've written so far I've been using 6 registers: EAX ECX EDX EBX ESI EDI.

Another suggestion i have is to make the subset of Intel a bit more regular; in other words, tweak the intel architecture to fix the not-quite orthogonal instruction set. Add memory to memory moves for example (something so wonderful in the Motorola 68000 instruction set), and maybe offer more registers, etc. When Prof. N. Wirth made his Lilith computer, he created a compiler that generated code for the National chip, the Motorola 68k and the intel chip, and his code generator was smallest for the national chip, second best for motorola, and significantly bigger for the intel chip. # Stefan Lesser 2019-10-22 08:32
Hey @Kartik Agaram, I finally got to read your two published blog posts about mu. They were very useful to learn about what you are working on and also to understand your values. I like your focus on simplicity. Or maybe it's minimalism?

I couldn’t help but think of MLIR (https://github.com/tensorflow/mlir#mlir-talks), which seems strangely similar and different at the same time. You have very specific ideas about each of the three layers you propose. MLIR is obviously much more complex because its goals are different (e.g. optimization), but it is also fundamentally about layers between the machine and a high-level language. And it’s about multiple layers (which is how it differs from LLVM).

Mu also reminds me of http://scheme2006.cs.uchicago.edu/11-ghuloum.pdf. Again layers, here much more granular than what you’re up to.

I have also been fascinated by the idea of “cleaning up” the stack, remove all the cruft we don’t really need anymore, and design a more sleek and minimal base system that a single individual can comprehend completely. Much like in the spirit of Alan Kay's VPRI STEPS project.

A friend of mine is working on an interesting project that is also in a way related, however focusing on the past to remove complexity: https://www.pagetable.com/?p=1373

It's tough for a single individual to build the whole stack, especially for today's hardware, even if you figure out an infrastructure that will save you so much time and effort to build things on — the figuring out part is what takes so long.

But if you can’t build the whole thing, then it's difficult to decide on which side of the spectrum to start. You (as many others) chose the low-level, bare metal side, and build it up from there. For sure that has potential to make these lower levels so much more effective. But it's a long way to get to a level that application programmers or even “end-users” will want to touch at all.

If you start on the other end, you can build pretty much anything on any hardware and focus on interaction, but you buy that flexibility with dependencies. And I haven’t seen many projects starting on that side tackling the dependency issue by methodically working on removing or replacing such dependencies. That could be a sign that figuring out the interaction is hard enough to never get to the cleaning up dependencies part, but frankly I think many developers are fine with the dependencies and probably don’t think about it too much. The web is often a good platform choice where you can make yourself feel better about dependencies “because they are (based on) open standards”. This never really worked for me (and clearly that's my problem, not the web's), but I doubt we get to a great place by piling onto a massive stack that gets more and more difficult to understand and maintain.

Ah, well, not sure where I was going with this… I enjoyed reading your blog posts and they made me think about this. So here you go. Not sure if that’s really useful feedback.
