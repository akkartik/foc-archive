# Chris Maughan 2020-11-13 04:30
Towards the end of the video I show a profile of the sample music. For comparison, here's a single frame of the UI on a release build. The UI is locked at 60fps; so can't go faster than that, but you can also see a couple of audio frames on this screenshot (which don't necessarily align inside frames), along with the FFT thread at the bottom.
Here 512 frames of audio is being generated in about 1.2ms - this is the equivalent of about 11ms of audio. Although I keep an eye on this stuff, and build things sensibly to avoid bottlenecks, I haven't actually spent too much time optimising anything yet. But it's good to know there is headroom.
I also forgot to mention in the video that one of the main things the profiler brings to the table is debugging; it already showed me a place where I'm doing unexpected work, and a mutex lock that should not be happening. For the small effort it took to build this feature, I think it was well worth it.

# Kartik Agaram 2020-11-13 07:16
The profiler looks awesome. Is it intended to only be a debug tool for yourself and anybody hacking on the code rather than making music?
# Chris Maughan 2020-11-13 07:57
Yes, I think so. Though I can imagine other uses for it. You could possibly imagine a simplified version which shows the live coder which parts of their performance are taking the most time. That's probably just looking for a hammer now I've got the nails though ;)
Also, I think I will add frame candles so you can see spikes in performance and indications of memory allocations. I'll put these things in as I need them.
I looked for an ImGui profiler before doing this one; I found one that didn't handle threads very well, or profile events that spanned frames. I found another that was a butchered version of the first one, with better UI but worse profiling. In the end, this only took a few hours over the course of a week to do; I'm pleased I put in the effort. My version will be OS in a week or so when I've finished tweaking it and pushed it to my shared library.
I'm not claiming this is fast, but for my purposes it is 'fast enough' :slightly_smiling_face: It allocates all the memory up front, and records simple time stamps, so overhead is fairly decent, which is all I need.
