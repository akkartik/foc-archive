The current state of software is a mess. The incidental complexity is unbelievable. Nothing changed in the past 40 years. We must have reached a local maxima.

Here are some of my heuristics for the future of programming:

- Using software and creating software should feel the same.
- 80% of people should be able to program without any formal training.
- You should be productive when programming on a 5" phone (but the optimal interface should be VR or R).
- Programming should not involve text files and syntax.
- Programming by example (with direct manipulation) should be the dominant paradigm (WYSIWYG, TDD, ML dataset, Excel's Flash Fill).
- The distinction between natural language, programming language and graphical user interface should disappear.
- Programming is mainly a modeling problem.
- Designing and implementing GUI by hand doesn't scale (especially once responsive design considers all of a user's preferences) and should be automated (e.g., Naked Objects).
- The code to invoke a user (GUI) and the code to invoke a machine (API) should be identical.
- Every function/predicate/model ever created should be instantly available to everyone.
- Mechnical turks and AI functions should be as easy to call as a local function.
- Fuzzy/bayesian values (e.g., "integer between 0 and 18") should be first class citizens, and automatically turn any deterministic functions into a fuzzy/probabilistic functions.
- Strings should be avoided, and semantically richer types/ontologies (Email, PhoneNumber, Password, Country) should be used instead.

Right now, I'm experimenting with Quantified Self. I'm trying to collect as much information about my life as I can. This starts with IoT devices, wearables, NFC tags, QR codes, cameras, audio recorders, GPS trackers, keyloggers, IFTTT, manual input, etc. Then I have to manually/automatically classify/quantify the collected data, to extract computable meaning/semantics. Ultimately, I can visualize the data, query it, infer new knowledge, make lifestyle changes to affect the output, etc. The idea is that the more we know about a user, the less we need to ask it for input ("don't make me click") and the less assumptions about it we need to hard code in our code base. I believe this is the first step to make coding against a user as easy as coding against an API or a database.

I'm looking forward to some great discussions!
