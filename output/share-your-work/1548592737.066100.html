# Gregor 2019-01-27 04:38
Alrighty, I need the rubber to hit the road: My editor is still in a bit of a rough shape, but I'd really appreciate getting some early feedback:
The basic idea is having a code editor that's fairly close to the text based editors we have atm but that uses more contextual information to make every keypress more meaningful . "Every" being the eventual goal, in its current state the right word would be "Some".
Similar to my previous project, Snytactor, it's atm only a JS editor. Supported syntax currently includes: declarations, arrays, objects, arrow functions (somewhat), function calls & if/for statements. Happy to get comments / bug reports / ideas or even PRs :upside_down_face:
https://gregoor.github.io/tofu/

# Felix Kohlgrüber 2019-02-05 01:10
I really like the sidebar that shows possible actions depending on the current context. This makes exploring the tool much easier. But I also agree with Steve that a video / gif showing the basic functionality would be helpful.
I did some experiments with structured editing myself last year (https://github.com/fkohlgrueber/vrite), but didn't get that far...
I think that one reason structured editing hasn't become popular might be that most structured editors are tightly coupled to a specific language. People use different languages all the time and learning new interactions for each language would be too much effort. It would probably be interesting to try to develop interactions that work well with all kinds of languages.
# Gregor 2019-02-05 08:03
Thanks, I hope I find some time to make a video next week (dayjob keeping me busy :see_no_evil:). And great to see that your exploring that space in a similar way!
RE supporting different languages: I agree that ultimately such an editor should support multiple languages, but I don't think that's the reason why it hasn't gotten popular. My theory for that is that few structural editors focus enough on the editing experience. Imo the only one that got close is Paredit (unfortunately they solved it for a niche-y language). My masterplan (as Steve called it in the podcast I just listened to) would be to prove it first for a popular lang, like JS, and use the learnings from that to build something more general purpose

# fyr 2019-06-02 11:38
looks great! here are some things that jumped out:
- The first GIF concerns navigation by word; probably for similar reasons, I usually navigate code(and prose) by word with alt-left/right instead of just left/right. However in tofu, alt-left/right is bound to… moving lines up and down? This seems like it might be desirable to delegate to cmd-alt-up/down or an alternate mode or somesuch
- the smart select seems somewhat aggressive: if the cursor is at the beginning of sketch.noStroke();, shift-right successively selects s sk ske sket sketc sketch
sketch.noStroke();
sketch.rect(0, 0, sketch.width, sketch.height);

- the smart select seems not aggressive enough: if the cursor is at the beginning of 0, 0, shift-right successively selects 0 0, 0 0, 0, sketch.width 0, 0, sketch.width, sketch.height [nothing happens, you have to press shift-up to select the closing paren and corresponding .rect(...)]
- to be precise, there is actually no way to select just .rect(...), only sketch.rect(…);, but subsequently shift-right does work to select lines
- when typing these up, I tried to select things with the mouse to find better examples, but while the editor shows a mouse selection preview this immediately collapses to the selections-start cursor at mouseup. While properly normalizing freeform selection is probably a challenge in itself, would it be possible for the cursor to just travel along the mouse when dragged instead of giving the misleading preview? (and jumping back to the initial click instead of where the mouse is let go, if the selection is forwards instead of backwards)
