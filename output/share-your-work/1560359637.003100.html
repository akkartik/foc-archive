# Edward de Jong 2019-06-13 01:33
I don't see too many flaws at first glance. The big one that stands out is the usage of try/catch, which is a horrendous way to handle errors. You are going to have a ton of errors in your system, where the things you are connecting to are going to fail to respond in a timely manner. Intrinsically, these services are finite state machines, yet you have no syntax for FSM. This is a huge omission, and will come to haunt you later. try/catch is not going to deliver the necessary granularity to debug flaky interactions.

I would suggest elif for chaining IF statements as the extra typing is unnecessary. I would suggest using NOT instead of !, and <> for not equal, as i do in beads as the exclamation point is one of the precious punctuation marks you might want to use later in your evolution.
# Edward de Jong 2019-06-13 01:35
As another minor note, it has proven dangerous from past experience to have a beginning block delimiter be identical the ending block delimiter. So ### should not be start and stop, because once you delete one of them, you have now flipped the meaning of what is commented versus not commented. So C uses / ... /, Modula-2 uses (... ), you might consider >>> .... <<< or some other symmetrical but not identical delimiter.
# Edward de Jong 2019-06-13 01:40
And by the way when you click "debugging" in the docs you get a 404 page not found error. How you are going to debug these scripts is one of the key aspects of your system, since 80% of the time spent programming is consumed in the debug phase, and your language will be no different.
# Edward de Jong 2019-06-13 01:44
nice to see a time fundamental type, however, it appears that it doesn't cover calendar dates. I think this will cause immediate problems, as many business processes are tied to calendars, so you either need to extend time to include a date+ time, or add a date data type. The current unix cron syntax is horrendous, and i am sure you can improve upon this easily.
# Edward de Jong 2019-06-13 01:48
for block comments if you want to stay with # as the starting punct, you could use #>>> ..... <<<# for example
# Edward de Jong 2019-06-13 01:50
In the Beads language we offer 2 block comment syntaxes, and support nesting, because commenting out a big block of code that has comments inside is painful if nesting is not supported. C has this problem, and it is huge. We offer 2 different block comments because it is helpful when trying to find the end of a block if you only use a special one for big runs of code.
# Edward de Jong 2019-06-13 01:51
I could go on for hours about all the subtleties of supporting comments, and how it interacts with computer generated code, but one should always make it easy and convenient to put in comments, because lazy people don't do it. Comments are for the humans reading the code, the code is for the computer.
# Edward de Jong 2019-06-13 01:54
There is an abhorrent trend where relatively inexperienced programmers are now proudly stating they refuse to comment. If they had to work on some 300k line program left behind by sadists, they wouldn't be so proud of their lack of commenting. It a curse on future generations to do anything than your best effort to make it easier for the next person to understand your program.
# Edward de Jong 2019-06-13 01:59
I am not a fan of arrays based on 0. This is a hangover from C where an array was basically nothing more than an alias to a pointer, and a zero offset was the first element. There is not a human being who is taught to count starting with 0. If you tell a kid to count to five you go 1, 2, 3, 4, 5. So not sure why you are going to continue this unnatural practice of 0 based. Your language is interpreted. In Beads we start at 1 by default, however arrays are sparse and thus you can have negative subscripts, not to mention enumerated type subscripts. You probably don't need that kind of power, but since your language is targeting a very high level, it is incongruous to my thinking to inherit an assembly language level piece of thinking.
# Edward de Jong 2019-06-13 02:02
It is also inevitable that you will need to support breaking out of a nested loop, so you will need to have optional loop labels for that case where you want to continue the outer loop, or break out of nested loops.
# Edward de Jong 2019-06-13 02:04
i also think the end command should probably have an optional error return code, as sub-scripts may fail in different ways, and being able to detect the kind of error will be occasionally needed. A script is like calling a function; it may need to pass information back up the chain.
