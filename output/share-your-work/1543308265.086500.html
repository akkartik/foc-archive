# Duncan Cragg 2018-11-27 09:10
Hi @Joshua Horowitz

I think many of us here are on-board with the concept of being able to (finally!) see the data that we're "animating" with our functions or behaviours, so :thumbsup::skin-tone-3: to that! And so I like the corresponding switchover from functions to data as the primary entity on the UI, compared with other box-and-line IDEs.

Now, true logic nerds may have something to say here, but to me it seems that this system may not be Turing Complete, because it looks like you only allow pure functions between data items. This, if true, is missing a critical piece of functionality: the ability to remember past states. As I understand it, this would require either recursive function calling, or circular loops, in your diagrams. I would suggest adding circular loops, not recursion, because recursion is hard to understand and visualise.

On the subject raised by @Vladimir Gordeev - I/O: you're not far from having it! You do have something rather I/O-like in the non-pure functions that call out to the remote API service and that call a randomisation function. You also have I/O in the output visualisations: you could imagine building a webpage from rendered forms of output data. And you can play with example values with the mouse, which is kind of like inputs. All you need is to wire in mouse position, buttons, textfields, etc, as your left-hand values (e.g. button up, button down as a boolean input value) and you have a completely general programming system, assuming you have added the circularity feature!

# Joshua Horowitz 2018-11-27 12:17
@Duncan Cragg Thanks for your thoughts.

Re: Turing-completeness – The system allows recursive function calls. For one example, go to http://joshuahhh.com/projects/pane/demo/ and press "square roots". There's a function there called "auto-improve guess" which calls itself, to iteratively find a good-enough square root for a number. You can press the little "[+]" to expand out the recursive calls, which is fun, but pretty unwieldy. As you mention, I think providing a visualizable "loop" construct would be more useful, and in particular I think tail recursion is a good foundation for this.

Re: I/O – The function which calls out to an API is a total hack, and if it had an actual side-effect, or depended on sequencing, rather than being an quasi-pure function, its behavior would be totally unpredictable. Anyway, my present stance is that PANE just isn't suited to making stateful, interactive systems, since it doesn't have visualization schemes (like timelines!) suited to seeing the behavior of stateful, interactive systems. You could try to shoehorn them in, but I think you're better off making a new interface for that kind of stuff.

# Duncan Cragg 2018-11-27 14:02
Turing completeness and loops

The pure FP solution that Pane would naturally fall into is what I see as plate-spinning: keep the functions calling and pass evolving, transformed state down and down or round and round, probably like you say with tail recursion, stream-based data types and/or Monads (whack me gently if I'm getting my Computer Science wrong here..). This has the problem that to visualise the nested state becomes very hard, as you said and as you showed in the square root example (I don't see how tail recursion really helps there, but I'd like to see your thoughts).

The not-fully-pure-FP solution (e.g. Onex) allows state as a first class concept and keeps the state "up at the top" where you can visualise it more easily. It can do this by allowing looping or circularity. Instead of the result of a pure function application (or the arguments of a recursive call) being new, transient, anonymous states in the call stack, your pure functions can take top-level first class state values as input, and produce new values as output, back in the top-level state they're associated with.

This makes visualisation much easier - and state chunks, being longer-lived and stable but evolving, can now be given unique identifiers for reference by other chunks.
