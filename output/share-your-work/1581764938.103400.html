# Ope 2020-03-03 05:53
@Cole Lawrence sorry I havent been here a lot.

Yes, one could specify a subset of Json schema and just go with that but that’s not a real fix imho.

There is things it does like dependent keywords (properties, additionalproperties and patternProperties depend on each other) and types that change (e.g a type takes a list or a string) that I think are problematic and too complex to understand(if you read the latest specification I think you would agree - stuff like recursiveRef, annotations etc).

One of the reasons for doing these things is better developer ux. So the canonical example they have of using writing a schema - {“type”: [“number”, “string”], “maxLength”: 4} . this validates a string of length 4 or a number (maxLength applies to just strings).

All of these kind of problems disappear with a DSL. You might ask why a new language? but i think even though json schema is written with json, you need to understand it is semantics for it to make sense just like any other language and having one designed from the ground up to support validation is better + you can always compile to native languages just like protobufs for example.
