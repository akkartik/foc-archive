# Will Crichton 2019-01-31 09:53
The only concern I have with this approach is that while this is certainly sensible for a software engineering class, a programming language theory class has different requirements. Such an assignment would need to not just show how to solve bugs in the real world, but to demonstrate how specific features of programming languages (and ideally type systems) are uniquely useful in making the refactoring process more productive.

# Duncan Woods 2019-02-01 02:39
Yep. The idea of separating "software engineering" and "programming language design" as different subjects is kind of horrifying. Design has to be informed by studying use exactly as you are doing.

I don't think a lot of the claims about programming language features e.g. advanced type systems and purity are evidence based. I think we need to rebrand PL design away from theory and mathematics to something pragmatic based on data from observations e.g. call it "PL Engineering".

Size of project is definitely a factor for understanding language features. You just don't face problems that matter in primitive languages on toy projects. There is a hierarchy of comfort something like:

100 LOC - asm
1000 LOC - dynamic types
10000 LOC - static types
100000 LOC - modules
1000000 LOC - package managers

I won't experience much pain writing up to 1000 lines of a dynamically typed language but I'd refuse to write 100k+ lines without static types, generics, lambdas, refactoring tools, profilers and static analysis tools etc.
# Duncan Woods 2019-02-01 02:48
An idea for an exercise that might be fun:
- have a 3 hour work shop to build a little game
- everyone starts on their own machine
- every 15 minutes, a whistle blows and they have to get up and move to the machine on their left and continue building the game based on their neighbour's progress
- everyone reviews the history of each project to work out why some projects proceeded faster than others

Might learn a little bit about language design and readability / safety but more about engineering practices. I think students would quickly invent a lot of strategies for how to leave incomplete work e.g. inline documentation about what they were aiming to achieve.
