# Janne Aukia 2019-01-10 01:00
Based on my work experience as a UX designer, I feel that there is very strong split between different types users on whether they prefer coding or not.

Some people have very strong wish to be able to type raw code (SQL, DSL, Lua, etc) and get frustrated with UI-based tools. If there is an option to either use UI or code, they always check the code first.

While others are scared of typing even a single line of code but can do very complex (possibly even Turing-complete!) configurations using a UI. But if something they perceive as coding is required, they ask somebody else to help.

This is definitely something to take into account when building a tool. The UI should be based on the types of users you have for the tool.

# Gregg Tavares 2019-01-14 06:05
I don't know postman that well. As a coder I find there are tons of repetitive tasks. A good editor helps me with those tasks. I can copy and paste from other code. I can search and replace after I've copied. I can regex search and replace or keyboard macro search and replace. In other words, the fact that it's text is a force multiplier.

Whether that fits users of postman I have no idea.

I mostly come from games and an example of frustrating for me would be for example a typical 3D program vs raw code. In draw code I can type


[
   "images/playbutton.jpg",
]


And then as new buttons are added I can just copy and paste and edit


[
   "images/playbutton.jpg",
   "images/pausebutton.jpg",
   "images/rewindbutton.jpg",
   "images/fastforwardbutton.jpg",
   "images/stopbutton.jpg",
]


That literally took 10 seconds to edit. Fast enough I it didn't interrupt my flow typing this comment. In many UI environments that ends up being 4x to 16x the work. Pick "import image", dialog appear, select image, set a bunch of manual settings, repeat, etc. And, even after I've managed to get all my stuff in the editor I then have to hook them up.

Random feedback (which probably is not useful but going to throw it out there)

Maya (the 3d package) has a feature that you can select any number of objects and then pull up spreadsheet view showing all their properties. Each row is an object, each column is a property. If one object hasn't have that property it's cell is marked in some way.

It's in this way you can quickly compare a bunch of properties across object as well as edit them.

I say that's not useful because I think, though I'm not really a Maya user, that not that many Maya users use that feature. I guess I'd have to go ask.

I mostly just bring it up as one example of a way to help power users with UIs that end up getting in their way more than helping.
