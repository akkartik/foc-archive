# Doug Moen 2019-06-19 06:08
APL is not a domain specific language. To use it effectively, you need to learn how to encode your domain using APL data types and operations. For example, Aaron Hsu has created co-dfns, an optimizing compiler for APL that generates GPU code, in about 750 lines of code. The compiler itself runs on a GPU. My own GPU compiler is >16,000 lines of object oriented C++, with only a rudimentary optimizer. I would not have thought it possible to write this kind of software in APL, or to run a compiler on a GPU. The compactness of Aaron's code is astonishing. https://news.ycombinator.com/item?id=13797797

# Doug Moen 2019-06-19 06:23
K has a seemingly richer set of data types than APL or J: records and atoms are a nice addition. The K data types are pretty close to JSON, so if you can encode your domain data in JSON, you can also encode them in K. I'm not actually an expert in APL/J/K, but I see a lot of potential in the basic design.

# Edward de Jong 2019-06-20 00:52
APL was decades ahead of all the other languages of its day. It was very close to what Mathematica became. It had a REPL terminal environment, and there was a built-in plot function that drew a nice graph (even on golf-ball terminal). Languages which have graphing built into the primitive system are wonderful for getting projects going fast. This is why Excel is so commonly used, and perhaps a very strong reason why Tableau got picked up for such a huge valuation. Drawing a nice graph is constantly needing to be done, and having such facilities already at hand makes for a much easier product acceptance. APL was actually a pain in the ass, and a few weeks after writing a paragraph of code it was like cryptography to read it again.
