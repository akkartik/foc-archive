# Dan Cook 2019-09-27 23:53
My personal project is about blurring the lines between code, compiler, and running program; code and editor; programming and user interaction; programming language and regular old program abstractions; etc.

I've found it hard to explain what/why/how, but I just realized that I can demonstrate a lot of this by writing the same program several different ways:

  • I want to simulate a digital circuit, but (let's pretend) no good tool exists, so I write a program that does it all from scratch. The simulator and the specific circuit all live in the same program / code.
  • Similar, but split the simulator into an external library that I use (but cannot modify) to build my circuit
  • Write a "programming language" to specify the circuit, and an "interpreter" that runs the simulation
  • Similar, but a compiler that generates the code (which in turn closely resembles the first approach)
  • A structural code editor instead of passing textual code.
  • A visual editor (e.g. editing a circuit diagram) instead of a programming language, but it's the same underlying structure as before, which still goes through an interpreter or compiler
  • Similar, but the simulation is live as you edit - there's no compilation "pass"
  • Similar, but also have the ability to generate code for it. (The analogy is having "code" be something live and perhaps non-textual; but "real code" is necessary in existing professional environments, you still build it -- but that doesn't mean that's what you have to "program" with)
  • Compare the limitations / ease of "changing the rules" for each different approach (compare what's involved to make the same kind of change for each approach)

    This will be easier to get up and running than my main project, and be a better demonstration / analogy for what I'm after (e.g. replacing "programming language" with something fluid that your code can change & control, rather than being subject to it -- a whole new level of something being "written in itself")
