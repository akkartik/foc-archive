# Pablo Donato 2026-01-05 07:37
A bit harsh for functional programming (as already stated by others), but it summarizes nicely my own feelings toward the current crop of (mainstream) programming language research. We shouldn't dismiss type theory altogether, but should definitely stop with the oversophisticated type systems, and should rethink over the computational substrate/notation they are founded upon.

I think there is a lot of interesting work in this space with string diagrams, if one still wants the mathematical beauty and rigor. But we lack a standard way to represent blackbox abstraction, which is really the purpose that we force upon and expect from functions. We also need more « science popularization » to get this out of the hands of mathematicians who focus too much on (denotational) semantics, and into the hands of engineers designing languages for real-world computations. Might take years or decades though, notably requiring some huge innovations in visual programming environments (text-only is too limited IMO).
# Paul Tarvydas 2026-01-05 08:22
In The Computer Revolution Hasn't Happened Yet Alan Kay says "...problem finding is more important than problem solving...". A problem, as I see it, is "software doesn't just snap together like LEGO blocks". Carefully fitting gears together isn't the same as snapping parts together, IMO. If that's the problem, then one needs only to make a list of the reasons and to systematically solve each of them. I think some of these are:
• synchronous behaviour (this is the definition of the synchronous/sequential programming paradigm, inspired by sequencer ICs (aka "CPUs"), not ideas like the internet or robotics or IoT or etc),
• routing (FP dictates one and only one routing policy - sequential, blocking),
• fan-out (FP forbids it, UNIX doesn't support it (you have to explicitly add "tee"), FBP forbids it),
• lack of recursive definition of software units for simplification by abstraction, over-complicated types for input and output signatures,
• lack of layering of types (I can't think of what a good phrase/word for this is, something akin to the OSI-7 layer model for networking and wrapping data in a Russian doll fashion (simple transport layer on the outside, more complication as successive layers are peeled off))
Many of these problems have been attacked by force-fitting baubles into the functional paradigm. Now that it has become easy to build new syntaxes (PEG, OhmJS, drawio) I favour using a hybrid of languages instead of stretching the functional paradigm even further out of shape. I have lots of working examples of software built this way, but due to my interest in language design, many of the examples are kinda heavy. Maybe the simplest example is a 3-part LLM thingie I threw together on a whim (that feedback loop is not "recursion"). Every implementation of Actors that I've seen has been polluted by the functional mindset. I find the PBP manifestation to be friendlier.
