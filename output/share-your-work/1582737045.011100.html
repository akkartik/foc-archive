# Sol Bekic 2020-02-29 02:55
@Kartik Agaram thanks, will apply :)

@Thomas Ballinger: It might be more powerful, but I don't think in metaprogramming: as I understand, RxJS runs JS to create the expression tree. So basically you have a turing-complete preprocessor. I don't have enough primitives for eval-time turing completeness, but that should just be a matter of time as far as I can tell. The differences to Obervable as far as I can tell are two:

1.) My language has explicitly defined semantics and UX for updating the graph at runtime. This is the main contribution I think, I'm not expecting to come up with a better general purpose FRP language here.
2.) The metaprogramming and flowbased logic have the same syntax and share some semantics. The metaprogramming only runs at eval-time, but at eval-time a single tick of flow-logic is also executed incrementally as the tree is built. Due to the dependency tracking, it is also known whether values are dependent on the root event or not. Based on this, the metaprogramming operations can access computation results, and determine whether values are subject to future changes or not. So (+ 3 3) is an eval-time constant, while (+ 3 (time/count 1)) is not (time/count has an implicit dependency on the root event). It's essentially constant folding.

tick is not really time dependent, it gets triggered whenever any of the input streams are 'dirty'. This is what I'm fiddling with currently; there is a kind of dichotomy between 'event streams' and 'value streams' and I'm figuring out where that separation should be made (is it the value type? is it a style of reacting to changes? is it a style of output stream?)
# Sol Bekic 2020-02-29 07:47
event streams are 'dirty' even if the value is the same as the previous one (e.g. playing the note C then some time later the note C again - two events). Between events the stream is 'empty'. On the other hand value streams are always at a value, and may even want to be interpolated. They don't need to be marked dirty when the new value is the old, and reassigning an input from one value stream to another shouldn't be considered a change when the values are the same either.
