# Sol Bekic 2020-07-15 11:01
@Jimmy Miller yeah, it's a touchy subject ;)
Since Atom renders as a html page, you can do all kinds of things (adding DOM, changing styling) etc, but the hard part is staying within what Atom can handle to make the editing experience still work. For example you could add or remove content with these featues, but then clicking at one location places the cursor at a different one and similar things, so you can break the editing experience really quickly.

If you pay close attention in the moments where I toggle the live view feature on and off, you can see that all my widgets take the place of the [tags] that every expression has. This way I can get rid of them and also reserve space for the widgets :wink:
# Sol Bekic 2020-07-15 11:03
I'm using a tree-sitter parser in Atom to parse the source code into a simple AST, then I get the Ranges that different elements occupy and turn them into "Marker". The Markers can be "decorated"; I use three styles of decoration:
- type: text with color: transparent to make the tags invisible
- type: overlay with a custom DOM node to fill the custom widget into the newly created gap
- type: highlight to highlight the background behind active elements in (switch) etc.
# Sol Bekic 2020-07-15 11:06
@Jimmy Miller most of what you are interested in is in this file: https://github.com/s-ol/language-alv/blob/editor-support/lib/highlights.js

Atom also knows "block decorations" which are more or less the only way of adding larger and varying-size pieces of custom information to the editor. They can be placed only between lines of the editor and can show whatever they want at any height they choose there.

# Charlie Roberts 2020-07-19 10:07
@Sol Bekic it’s not perfect, and the parsing is a nightmare because .js has so many edge cases, but it “kinda mostly” just works. Most of the annotations are css on top of the actual code, so you can edit away no problem. If annotations do get broken during editing—for example, if you completely delete a token and insert a new one the previously associated CSS is also deleted—they are usually correctly restored once the edited line of code is evaluated again. The inline visualizations can be deleted / skipped over with regular key/mouse gestures just like text, but currently don’t get copied to the clipboard. Curious if you looked a CodeMirror… I do think it handles many of the use cases you just listed.
