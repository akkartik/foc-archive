# Stefan Lesser 2018-09-28 04:01
Some random thoughts after reading it once:

1. Weren’t the first computers also more like the Smalltalk system then, when we still set state directly, before being able to read in serialized programs from punch cards, and later (paper or magnetic) tapes? The point being: maybe the serialized aspect isn’t as important as a distinction?

2. I see what you’re doing to prevent premature definitions, but ultimately just “system” feels to unspecific. Everything is a system these days. Are we talking about interactive (non-)deterministic systems here? Do these properties help us to better define what we’re designing? Surely bringing up UI locks you into a more specific type of system. But maybe UI doesn’t have to be involved and are we then talking about a different kind of system?

3. I always felt that there is a step we take when we move from a program that is basically just a more complex function (start-input-calculate-output-done) or collection of computations to something that is based on a core loop, which is sometimes also called a simulation. There is certainly a difference between these and it might or might not be important for what you want to get at.
To me it feels we left “programs” behind a long time ago and misuse the name for things that aren’t really programs anymore, and that’s just confusing everyone.

4. I wholeheartedly agree that this is about “designing systems” and that “programming” as in modifying a serial representation of instructions with symbols is just one way of designing these systems.

# Shalabh 2018-09-28 13:29
@Stefan Lesser - great points.
> “system” feels to unspecific. Everything is a system these days.
I agree. Even a program can be thought of as a 'system'. I struggle with how to phrase this better. I could start with 'just physical hardware' but I do want to include the idea of 'software', even while not adding any notions of what the software looks like.

> I always felt that there is a step we take when we move from a program that is basically just a more complex function (start-input-calculate-output-done) or collection of computations to something that is based on a core loop, which is sometimes also called a simulation.

Interesting.. the later model can simulate any system so it's a virtual system in that sense.

> a more declarative/functional way of describing that difference between program and system as I defined it (probably wrongly) would be?

I struggled with precise definitions too and then gave up because these are all made up categories in the study of artificial machines. To me the system provides the idea of a program. The program is operated on from the 'outside' - you write it, run it etc. so there is other stuff outside of the program that lets you do this - these are parts of the system outside the program.

Instead of definitions, what I'm trying to identify are the typical connotations associated with 'a program' - e.g. the fact that it has two 'views': the static structural internal view in the programming language and the dynamic interactive run time view (in the UI language?). Another one is that whenever you run it, it starts afresh and the internal state is reset. Identifying these might indicate room to explore for whatever other abstractions we design.
