# Harry Brundage 2021-04-12 08:56
@em :wave: Mo's collaborator here. we're not actually running on cloud functions but on fission.io within our own K8S because it gives us a lot more control over cold start times which i personally loathe with a passion
# Harry Brundage 2021-04-12 08:58
and we currently don't actually provision any infrastructure for each app created -- it's all multitenant shared infrastructure right now which is either insane or brilliant, we aren't sure yet. i like the idea of the multitenancy because it lets you super elastically burst and really take advantage of the cloud-ness of a platform like us, you don't have to write any autoscaling code or bake any AMIs or docker containers, it's all done for you, including the database part. I used to work on Shopify's flash sale platform a bunch and I think it informed a lot of my thinking about all of this -- it's really cool for a platform to be able to offer individual tenants access to an unfairly large share of the platform's resources for the select few times they need them, and then not charge them for it all the time, it's something that only platforms can do
# Harry Brundage 2021-04-12 09:00
because of this though the apps that we run are tied to our infrastructurre pretty closely and aren't yet "exportable" in the sense that you could eject and terraform the same application into a cloud of your choosing. we felt that the coupling to a specific set of infrastructure that we run was worth it from a stability and capability perspective but i think we might lose a few folks that are really sensitive to vendor lock in until we have an eject button

# Cole Lawrence 2021-04-16 09:28
Cool demo @Mo! Great work @Harry Brundage & @Mo

The demo of state charts is really interesting (cc @Garth Goldwater @ 6m55s). Garth and I have discussed something similar for the product I'm working on at Storyscript, but instead of opening up to any state transitions, narrowing down to prioritize representing linear state progressions, as in:
Opened (Task Name) ->
+ In progress(Due Date) ->
+ Finished(Date Finished) ->
+ Accepted(Accepted By)
Where each transition requires the properties of the next state to be defined, and each state is additive of the properties of the previous states. My guess is that the majority of domain models will move in a "linear / acyclic" "additive" way the majority of the time. Another example is order fulfillment for a web store.

What do you all think? Do you all have any ambitious/out-there ideas for the future of state charts & domain modeling?
# Harry Brundage 2021-04-16 10:25
i think your instincts are right for sure, most things tend to go in those linear states. Mo and I both used to work at Shopify and have a borderline uncomfortably deep familiarity with the state machines powering the shopify order model, and notably there's actually two kind-of-orthogonal ones, one for the financial status and one for the fulfillment status. most of the time, merchants wait till they get all their money before sending out on order, but not always like with net 30 payment processing, or the other way around, they sometimes take the money way way before fulfillment is even possible, like with a kickstarter or pre-order
# Harry Brundage 2021-04-16 10:26
so we kinda think that while state machines are a really powerful way of modelling this logic, you still end up with a lot of extra transitions between states that happen rarely, but not never, and that can in fact be burdensome when using such a formal way of governing stuff
# Harry Brundage 2021-04-16 10:27
we hope that by owning the interface our developers are using to construct these state machines we can help with that and accelerate the creation of those not-super-frequently taken transitions, think like click and select 4 states then right click and hit a "create all possible transitions" menu option, easier to delete the ones you don't want than create all the ones you do
# Harry Brundage 2021-04-16 10:28
even for a kanban board backend, you should probably be able to move a card between any column at any time, and ya as you said just some of those transitions should require particular validations to be met before being take-able
# Harry Brundage 2021-04-16 10:30
one thing we haven't nailed yet is the relationship between those additive validations and the state that something is in, we have a facility for defining one global, always applying set of validations for each model same as Rails or Django or Postgres constraints might work, but i think you are totally right and there's a set of validations that only apply in the later stages of a record's lifecycle that would be nice to model as such
