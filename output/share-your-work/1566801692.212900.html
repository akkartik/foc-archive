# Brian Hempel 2019-08-26 11:39
> So when it crosses over the previous function, we must have a strategy for deciding which paths it travels without relying on history.
Probably impossible: unless you want to have a lot of fun doing it “wrong” on purpose! Consider the simple “plus” operator: given the output of a sum, how do “undo” the sum into its two input summands? There’s an infinite number of possibilities. Consequently, for example, in Perera et al.‘s “Functional Programs that Explain Their Work” (ICFP 2012, excerpted in screenshot), they record the original operands so you can undo the computation. Their goal is to rewind the computation in order to rebuild a minimal program required to produce the result, but the challenge is the same. (Their overall strategy is to use computation traces recorded during execution, the T‘s in the screenshot.)
# Brian Hempel 2019-08-26 11:48
Continuing with the “plus” example, the challenge is even greater if you manually change the output sum after it was computed and want to back-propagate to produce changed input summands. Turns out you can’t actually do this with local information only: you end up with a global constraint problem. Consider:

x = 5
y = 10
return (x + (x + y))

Forward execution produces the result 20. Say you manually change 20 to 21 and backprop, hoping to change x or y. The expression x + (...) evaluated last, so you change x from 5 to to 6 to make it 21 instead of 20. But now, suddenly, on forward execution the whole result is 22 rather than 21!
# Brian Hempel 2019-08-26 11:52
To get around this problem, in Chugh et al. “Programmatic and Direct Manipulation, Together at Last” (PLDI 2016) we record special arithmetic traces so that, instead of only seeing the local expression
x + (...)
we see an entire arithmetic expression
x + (x + y)
which the computer can change with simple algebra.

# Shalabh 2019-08-26 12:39
Consider the 'state of the world' as an input to a function and the 'new state of the world' as the output. If two values in the input map to the same output value, it is impossible to determine from the output alone which of the inputs was provided to the function. The same problem exists when the same input value maps to multiple possible output values (assumes some untracked input e.g. random number generator).

So to be reversible there need to be a 1:1 mapping from input state to output state. This means you can't reduce the information content of the system (aggregation, hashing etc.). Logging is one way to keep the information content intact - the information is accumulated in this log while it is being lost elsewhere. There could be other ways, e.g. each comet could carry provenance information (comet local log). The design effort here is mainly around minimizing redundancy - so you try to keep the minimal information content that will get you reversibility.

Considering the 'state of the system' point of view, I don't draw any strong distinction between 'reversible functions' (e.g increment) and logging or tracing. The new system state has enough information to uniquely map back to the previous state - the difference is only where this information is present.

# Ivan Reese 2019-08-27 14:36
> how do you return from functions?

There's a bit of a conceptual mismatch between what I'm doing (heavily inspired by dataflow languages, hydraulic and electrical schematics, etc) and what you do when calling a function in a Fortran-inspired language (where, I believe, the notion of "calling" and "returning" a function come from). In the Fortran-inspired lot, code forms a tree (the AST), but execution forms a graph (the call graph, the cyclomatic graph, etc, which I believe are all DAGs). In Hest, Max/MSP, schematics, etc, code and execution form a graph (and not even a DAG — a totally free graph). So in the former, a function call travels down a tree branch from call site to function, and the return comes back up the same tree branch. In Hest (etc), the call and the return are separate, so you're responsible for setting up both.

It might also help to know that... in Hest, when you have a point acting as a function, that point isn't itself the function — it's a reference to the function (which is defined elsewhere). You can make as many such references as you want.

> It's like whenever you call a function it will return with same value everywhere where that same function is used.

Yep, exactly. If you don't want that, you just don't do it — make a separate copy of the function point everywhere you need it, and only have 1 inbound edge and 1 outbound edge.

# Ivan Reese 2019-09-03 10:01
> How do you handle functions that take two arguments?

Still considering different approaches, in combination with different modes of time travel, different ways of storing state internally, to see what feels good and avoids perf cliffs.

My current favourite is what you mentioned: functions don't open themselves to input until they have data waiting at every inlet. If multiple pieces of data arrive at one inlet before other inlets are filled, we just do last-write-wins. If you want a queue instead, that could just be a special kind of edge. I call this idea "perching" because it's like the data point is perched on the lip of the function, waiting until the function opens up.

Another option is that data points "paint" the edges they travel along, leaving a permanent trail of their data value. When a function receives a data point at an inlet, it can accept that input and produce a result even if other inlets don't have a data point waiting at them, so long as those inlet edges have previously been painted with a value. This is sort of like a version of FRP-style behaviours / signals depicted as standing waves, where the initial wavefront is a point. It's also analogous to the way that variables in a scope work, which are in essence painted with a value as the instruction pointer passes over them to do an assignment and they remain so painted until the next assignment. (Yes, I do spend a lot of time thinking of our common abstract execution metaphors as little machines with agency which, I know, is a "no-no" in some technical writing circles.)

Painting seems really interesting and I definitely haven't explored it enough, but I have a raft of concerns that don't seem to be fading as I consider it more. Painting adds several new kinds of complexity (compared to perching and other ideas), and requires a whole additional set of editor tools. There might not be a good way for the artist to embody painted edges the way they can embody points — what would "painted edges: the game" be like, and is that useful for programming?

There's another problem that both painting and perching have, which I only really started to appreciate after exploring painting in depth. Currently, the act of a data point arriving at a function is two things combined together that might need to be separate — it's the provision of data, and it's the initiation of that function's execution. It's both (eg) a value and a "bang". I might want to separate these things, so that the system isn't "complecting what and when" to borrow a Hickeyism. (And that's 25¢ in the jar.)

And yes, of course, I am very resistant to any idea that crosses my path that'd lead to hidden state. Much like incidental complexity (which Hest warmly embraces), hidden state is something that will inevitably happen in some sense or another, but I want to be very careful about what things I hide, and how.

So, what do you think? Should I separate the movement of data and the movement of execution? There are lots of ways to do that. If you were providing multiple pieces of data to a function, how might you do it?

I could try to make the core of the language consist only of single argument functions that execute immediately when data is provided. This feels very.. lambda calculus. I can't conceive of how to make this performant in fuckin javascript. I think I have to do multi-argument, somehow.

"Synchrony nodes" is similar to an idea I've been toying with, which I call "sync gates". They're basically the "sync" operator in SpaceChem, used for temporal coordination — when a data point arrives at a sync gate, it sits there and waits until all other connected sync gates also have a point waiting at them, then all the gates open. No Y-combination, just little gate nodes connected with a "wire" edge. I though about using sync gates in combination with the idea that functions treat the simultaneous arrival of data points as a multi-argument invocation. But I think synchrony nodes are even better, because if you numbered the inbound branches of the Y, you could establish an order, which I think is essential. (I don't think I want to make Hest like a stack/concatenative language, using the temporal order that data arrives at a function as the argument order — that's too confusing).

It's hard for me to think about this stuff without also thinking about how an artist-defined function is created, so I should probably speak to that. When defining a function within Hest, you draw a graphic "icon" for that function using the vector art tools, and lay out the code next to that graphic. You also connect the code to the graphic as needed to (say) alter the graphic in response to state (because functions in Hest can be stateful). If you want to have (say) a row of inlet ports along the top of the graphic, you'd draw what those ports should look like, tell Hest to treat them as ports (which means they'll accept edge connections from outside), and route where the incoming data should go.
