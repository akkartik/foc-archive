# Robbie Gleichman 2019-11-25 02:01
I have been working on a compact visual representation for Haskell (part of the Glance project https://github.com/rgleichman/glance/). I'll be doing an in-depth video presentation soon, so as a sneak-peak, here's what FizzBuzz (https://wiki.haskell.org/Fizzbuzz) looks like.

This is the equivalent textual code:
fizz x | x mod 15 == 0 = "FizzBuzz"
| x mod 3 == 0 = "Fizz"
| x mod 5 == 0 = "Buzz"
| otherwise = show x

main = traverse_ putStrLn (fmap fizz (enumFromTo 1 100))

# Robbie Gleichman 2019-11-26 01:35
@Mariano Guerra

> the = on the clauses seems to be a star, but after main it's missing
This is intentional. There's not a 1:1 correspondence between the textual symbols and visual symbols. If fizz is written using MultiWayIf (which is slightly closer to the visual representation since Glance only has expressions, not declarations) the = in the clauses become ->

fizz' x = if
  | x `mod` 15 == 0 -> "FizzBuzz"
  | x `mod` 3  == 0 -> "Fizz"
  | x `mod` 5  == 0 -> "Buzz"
  | otherwise       -> show x

> in flow diagrams there's a standard symbol for alternative
Is this symbol a diamond? The symbols for the if expression (the half filled circle and the star) are fairly arbitrary, so most any two distinct shapes could work here.

> instead of wrapping everything in boxes, using parenthesis to delimit "boxes"
Great suggestion, I'll try this out.

@Joshua Horowitz

> Mathematicians have a notation for case-by-case analysis. [...] Is there a reason not to follow a path closer to this?
I think replacing the conditional symbol (half filled circle) with "if", and the value symbol (orange star) with "then" could work. One difference from the math notation is that oftentimes the value symbol is below the conditional. The bracket is problematic because it would cross over wires going left to right.

@Robbie Gleichman Instead of doing a fizzbuzz code example, which at its core is a mere 4 lines of code, why not tackle a larger task and see how you would represent it. # Robbie Gleichman 2019-12-08 15:30
@magicmouse94937 Yes, I agree that being able to represent large functions well is important. Although I do also think that there's value in being able to represent smaller bits of code compactly. One strategy for representing large functions is to decompose them into smaller functions. Compact representations of small functions makes this strategy more viable since all of the small functions can be fit close together on the screen.

@Yair Chuchem It's not more compact than the textual Haskell version. As an aside, it's interesting to ponder how to make a visual representation that is more compact than the textual version. Three ideas that come to mind are:
1. No variable names. In this example for instance all of the "x"s could be removed.
2. An understandable visual representation of point-free style code. I have not thought much about this since Glance needs to generate understandable textual code.
3. Use a 3rd dimension. For example space (AR/VR) or time (animations or user interaction).

# Edward de Jong 2019-12-20 17:45
You've done a great job exploring the various degrees of compactness. When you watch the morph sequence where you take the factorial function from its most exploded graphical form and take it all the way to textual, my mind instantly becomes relieved at the textual form resulting. I think for people well trained in reading code, these graphical tools seem like a step in the wrong direction. However for those not yet trained to read code, i would imagine the graphical form holds appeal. I would caution you that one thing you are not exploring is the effect of the code on the state of the systems. In some sense it is the runtime environment that is of interest as it executes. So instead of spending so much time evaluating all the different levels of compactness, perhaps exploring the animation aspects of the program as it runs would be more fruitful.
