# Breck Yunits 2021-01-16 16:44
Thanks for the feedback Konrad!

I would say at this stage of the game the 2-D/3-D aspect is
important for the UI of programming. It's like WYSIWYG of
programming.

In Tree, scope has a semantic meaning but also a geometric
meaning—if things are down and to the right, they belong
together.

In Tree, there aren't infinite ways to write a list. There's
just one, and it's the way you would do it in the real
world: put items next to each other.

The languages follow constraints imposed by geometry. While
this may seem like it would limit creativity, in fact I've
found the opposite. By letting you focus on signal, and
removing all noise, you can focus on the semantics, and
really simplify your code to its essence.

There are also lots more you can do when you break out of
1-dimensional, linearly parsed languages. Tree Notation is a
subset of Grid Notation. There are other 2-D notations
out there.

An example might help. Would be easier to do a visual
but could try to put it into words. Imagine a project
wouldn't be a folder with a bunch of files, but instead one
file that you would open with a spreadsheet like IDE. and
edit like a canvas. What you'd normally put into files you
would instead drop an "anchor" in some cell in the sheet,
and start typing. That would be a root tree. And you could
have many of them throughout the canvas, instead of 1 file.
This is a good structure for languages for simulation. There
are some things like this, but none of them (AFAIK) have
ever made the layout and code isomorphic, and when you do
that you don't give up anything, and you make parsing and
tool building a lot easier.

# Breck Yunits 2021-01-20 06:50
Good questions! Really helpful to see where the gaps are.
At the high level I view the big idea as "2-D Languages". (Well, 3-D
as well, but let's just limit ourselves to 2-D for the moment.)
I think of things all around me in the world as 2-D languages.
A Highway Sign for instance.

Once we get into the digital realm, with our current hardware
stack we do have to collapse things into 1-D, and that's
where "Grid Notation" and "Tree Notation" come in. Any document
could be parsed as either a Grid Structure or a Tree Structure. If
there are no nested nodes, then the structures would basically be the
same. If there are nested nodes, then in Tree Notation you would
parse child trees, whereas in Grid Notation the empty edge nodes
would just be parsed as empty cells. I rarely use pure "Grid Notation", but
I do sometimes use something in the middle, where sometimes I allow
nested trees and sometimes I don't. But really it's probably
better to just always use Tree Notation. Because even when I start a
lang small and think "I'm never going to need scopes with this
dumb config language", a few months later I'm always like "damn, scopes
would have been better and I shoudl have just used tree notation"
# Breck Yunits 2021-01-20 06:53
Honestly maybe it's confusing things more
than helping to talk about "Grid Notation" and "Tree Notation".
I just thought there's a nice pattern at work here:

Add 1 special symbol and you get a 1-d "List notation" that supports lists of words.
Add 1 more special symbol and you get a 2-d "grid notation" that supports lists of lists of words.
Add 1 more special symbol and you get a 3-d "tree notation" that supports recursive lists of lists of lists of words.
