# Ivan Reese 2019-10-09 12:57
Subpatchers and groups are basically the same thing, from a UI perspective. You select the nodes you want to group together, and then they're bundled up into a new composite node with all the required inputs and outputs.

The difference is how you treat the code inside the new composite node. In most visual languages, if you duplicate a "group", it makes a copy of all the code — if you make changes inside one copy of the group, that doesn't change the other copies of the group. Whereas creating a "subpatcher" in Max, or defining some sort of reusable node or symbol or function in other visual languages with this feature, means that changing the contents of one instance will change all the others — they're all just instances of the same canonical source code.

# Ivan Reese 2019-10-09 13:03
The group vs subpatcher distinction is just this:

When you make a new composite node that acts like a function/subroutine (whether you call this a "group", "subpatch", or whatever)... and you use a few copies of that composite node in your system... what happens when you change the composite node's internals? Are all the copies changed, or are they all independent?

I think that any truly great visual language would let you control that in powerful ways.
