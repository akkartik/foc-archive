# Jesse Aldridge 2018-10-23 14:58
Seems like there are three key components to building a general purpose system like this:
1. The user supplies a set of example inputs
2. Some computation done on those inputs
3. A visualization of the system is rendered on the fly whenever 1 or 2 changes

Number 3 seems like the hardest part by far in any such system. Although in theory you should be able to serialize the state of any system to a stream of text and then do additional parsing on this text to extract interesting information relatively easily (compared to producing rendered output on a canvas or something like that). # Kartik Agaram 2018-10-23 18:22
On a slightly different axis, it seems like you want to specify 3 things in your "workbench UI":
A) What to run (the component under test).
B) What context to run it in (like a reproducible test).
C) What to watch as you run it.

Once you have this set up (the loose feedback loop), you want to be able to make changes to A and instantly see them reflected in C (the tight feedback loop).

# Kartik Agaram 2018-10-24 11:10
I'd like to dispel any impression that I'm a debugger guy: https://news.ycombinator.com/item?id=11854319#11855294; https://news.ycombinator.com/item?id=11048633#11052363

Watches (and, I suspect, time travel debugging) are unused in gdb by pretty much everyone I know (even the debugger guys) because they're too slow. Some features just have too much impedance mismatch with the "debugger way". They'd be orders of magnitude simpler to support if you just dump a trace of an execution and browse it after the fact. You lose the ability to change programs when they hit an error. But I don't see too many people using debuggers like that either.
