# Felix Kohlgrüber 2018-10-03 14:35
I've lately been thinking that many problems in software development might be communication problems and not primarily related to the specific tools used. How often do people "reinvent the wheel" just because they don't know that there's a solution for their problem or because they aren't able to change a 90% solution to fit their needs? I'm thinking that changing the way people work together to produce programs could make a big difference. In particular, I find the idea of having a wikipedia-like platform for code very interesting. Some advantages could be the low barrier to contribute or that people could refactor similar parts of different projects to use the same general solution, therefore providing a "join" operation for the "fork" model that's common in OSS development. I've got more to say about this and I'll probably write a blog post about it, but before I do, I wanted to ask you guys if you know if anyone has done something similar before. Thanks :wink:

# Felix Kohlgrüber 2018-10-04 22:05
I'm glad that my post started a discussion. For me, the biggest difference from sth like github would be that anyone could edit stuff that others have created initially. Take documentation / tests for example: When you are writing a piece of software, you understand how it works, so documentation is mostly for making it easier for others. But if you just started, not knowing whether your program is going to be used by others or even if's going to be useful, writing documentation would slow you down without providing an immediate benefit for you. So people don't write docs in the first place. On the other hand, a user discovering a piece of code who has just figured out how it works is more likely to write some docs to make understanding easier for others. If you find a program without tests that looks promising, but you don't know if it actually does what you need, you'd probably try some examples to test it. Wouldn't it be great if you could easily contribute these tests to the program?

# Felix Kohlgrüber 2018-10-05 13:44
@Nick Smith the difference I find important is that there's another concept of ownership. Github projects are owned by their maintainers and they need to accept your pull request. In a wiki system, changes are applied without anyone needing to approve beforehand. I think this would work great for comments / docs, and for code (which needs more stability), each edit could lead to a new version with references to other code snippets also targeting a specific versions. Users would manually update to new versions, probably assisted by ratings or whether an update breaks relevant tests.
# Felix Kohlgrüber 2018-10-05 14:15
@William Taysom I share your daydream ;-) I think that incompatible and unmaintained projects are indeed a big problem and I think that a wiki-like system could address both. There's no unmaintained wikipedia site as long as anyone cares enough to perform changes. Incompatibility can mean a lot of things and I'm focusing on library incompatibility in the following (please correct me if you meant something else). If there are for example two projects for creating plots (like matplotlib and plotly for example), they would usually be developed by two different teams. Even if both projects shared a common core (e.g. drawing a simple line) that could be refactored, the project and team structures would most likely prevent this. In an optimal environment, anyone could search for common stuff and refactor it by replacing with a more general solution. This could evolve over time and lead to abstractions that a single person / team / company could not have come up with initially.
# Edward de Jong 2018-12-10 18:39
Fundamentally what Herr Kohlgruber is talking about is having software enter the age of interchangeable parts. This has not happened yet in the software field. It was pioneered by several inventors in the 1800's such as Eli Whitney, and was used to devastating success during the civil war, where it permitted mass-production of guns which previously had not existed. Interchangeable parts is indeed a worth goal of any next generation product, and i consider it a crucial criteria on which to evaluate any new programming system. We know from past experience that this is a tricky issue, as programs = algorithms + data structures, and data is something people just can't seem to agree on how to store. So it won't be easy.
